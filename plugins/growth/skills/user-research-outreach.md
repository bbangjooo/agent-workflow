# User Research Outreach — 1:1 유저 리서치로 위장한 초기 유저 확보

니치 커뮤니티에서 타겟 유저를 찾아 "유저 리서치"를 제안하고, 직접 설치/온보딩해주면서 첫 10명의 sticky user를 확보합니다.

## 설명

> "런칭하지 마라. 리서치하라. 첫 10명은 마케팅이 아니라 1:1 대화에서 나온다."

일반적인 아웃리치(write-outreach)와 다른 점:
- write-outreach = 인플루언서/기자/리더에게 **제품 소개** 목적
- user-research-outreach = 실제 타겟 유저에게 **리서치 참여** 요청 → **sticky user 전환**

핵심 차이:

| | 기존 아웃리치 | 유저 리서치 아웃리치 |
|---|---|---|
| 프레이밍 | "제품 써보세요" | "20분만 봐주세요" |
| 대상 | 인플루언서, 기자 | 실제 타겟 유저 |
| 목표 | 노출, 리뷰 | 설치 → 1주일 사용 → 유지 |
| 채널 | 이메일, LinkedIn | 니치 Discord, Telegram, GitHub, YouTube 댓글 |
| 관계 | 1회성 | 지속적 피드백 루프 |

## 입력

> **참고**: 아래 산출물이 존재하면 반드시 참조합니다:
> - `outputs/stage-0/idea-brief.md` — 제품, 문제, 타겟 유저
> - `outputs/stage-1/prd.md` — 핵심 기능
> - `outputs/stage-5/market-analysis.md` — 페르소나
> - `outputs/promotion/audience-language.md` — 유저 실제 표현

- 제품명, 해결하는 문제, 타겟 유저, 라이브 URL (또는 베타 링크)

## 실행 내용

### 1. 초좁은 페르소나 정의

```
market-analysis.md의 페르소나를 더 좁힙니다.

일반 페르소나: "개발자" ← 너무 넓음
좁은 페르소나: "AI 코딩 도구를 매일 3시간+ 쓰는 개발자" ← 아직 넓음
초좁은 페르소나: "Claude Code/Cursor를 쓰다가 번아웃을 경험한 개발자" ← 이 수준
더 좁은 예: "AI 출력 리뷰 ↔ 자기 코드 사이의 context switch로 인지 피로를 느끼는 개발자"

[초좁은 페르소나 설계 기준]
- 구체적 도구/플랫폼을 사용하는 사람
- 구체적 문제를 이미 인식하고 있는 사람 (번아웃, context switch 피로, AI brain fry 등)
- 어딘가에서 그 불만을 표현하고 있는 사람
- 10명만 찾으면 되므로 넓힐 필요 없음

산출물: 초좁은 페르소나 1-2개 정의
- 이름 태그 (예: "overcaffeinated AI dev")
- 행동 패턴 (무엇을 하는 사람인지)
- 불만 표현 (어디서 어떻게 불만을 토로하는지)
- 서식지 (어디에서 활동하는지)
```

### 2. 니치 서식지 탐색 (웹 서치)

```
타겟 유저가 실제로 불만을 토로하는 곳을 찾습니다.
범용 커뮤니티(r/startups, r/Entrepreneur)가 아닌 니치 채널을 탐색합니다.

[탐색 대상]
A. 니치 Discord 서버
   - 웹 서치: "{도구명} discord server"
   - 웹 서치: "{카테고리} community discord"
   - 예: "Claude Code discord", "Cursor users discord", "AI developers discord"
   - 수집: 서버명, 멤버 수, 활성도, 관련 채널

B. Telegram 그룹
   - 웹 서치: "{카테고리} telegram group"
   - 수집: 그룹명, 멤버 수, 활성도

C. GitHub repos/issues
   - 관련 도구의 GitHub issues에서 불만/요청 탐색
   - 웹 서치: "site:github.com {문제 키워드} issue"
   - 수집: repo명, 관련 issue, 활발한 참여자

D. YouTube 댓글
   - 웹 서치: "{카테고리} {문제} youtube"
   - 관련 영상의 댓글에서 불만 토로하는 유저 식별
   - 수집: 영상 URL, 댓글 작성자, 불만 내용

E. 니치 서브레딧
   - 범용이 아닌 도구/문제 특화 서브레딧
   - 웹 서치: "reddit {도구명}" 또는 "reddit {구체적 문제}"
   - 수집: 서브레딧명, 멤버 수, 관련 글

F. Slack 커뮤니티
   - 웹 서치: "{카테고리} slack community"
   - 수집: 커뮤니티명, 참여 방법

[탐색 결과 → 서식지 우선순위]
- 타겟 유저 밀도 (많을수록 좋음)
- DM 가능 여부 (직접 접근 가능한지)
- 활성도 (죽은 커뮤니티는 스킵)
- 자기홍보 수용도 (적대적이면 더 조심)
```

### 3. 리드 발견 & 프로파일링

```
니치 서식지에서 실제 타겟 유저를 식별합니다.

[발견 방법]
- 니치 Discord/Telegram에서 문제 관련 대화 참여자
- GitHub issues에서 관련 문제를 제기하거나 댓글 단 사람
- YouTube 관련 영상에 불만 댓글 남긴 사람
- Reddit에서 문제 관련 글/댓글 작성자
- Twitter에서 문제 관련 트윗한 사람

[각 리드별 수집 정보]
1. 핸들/이름
2. 플랫폼 (어디서 발견했는지)
3. 불만/니즈 요약 (그 사람이 직접 한 말 인용)
4. DM 가능 여부
5. 활동 수준 (최근 활동 날짜)
6. 적합도 (상/중/하)

[목표: 최소 15-20명 리드 → 10명에게 DM → 5-7명 응답 → 최종 10명 sticky user]
```

### 4. DM 메시지 — "유저 리서치" 프레이밍

```
핵심: "제품 써보세요"가 아니라 "리서치 도와주세요"

[DM 템플릿 — 영문]

"Hey {이름} — I saw your {댓글/글/이슈} about {구체적 문제}. 
I'm experimenting with something to {문제 해결 방법 한줄}. 
Can I watch you {작업} for 20 minutes and see where it'd fit? 
No pitch, just trying to learn."

context switch 각도 변형:
"Hey {이름} — I saw your post about {문제}. I've been tracking my own workflow 
and realized the constant context switching between AI output and my own thinking 
is what actually fries my brain, not the hours. I'm experimenting with something 
that detects that loop. Mind if I watch you code for 20 min to see if it's the same for you?"

[DM 템플릿 — 한국어]

"{이름}님 안녕하세요 — {채널}에서 {구체적 문제} 관련 글 보고 연락드립니다. 
저도 같은 문제를 겪어서 {해결 방법 한줄}을 실험 중인데요, 
혹시 20분 정도 {작업}하시는 걸 보면서 맞는지 확인해볼 수 있을까요?
홍보 아니고 순수하게 맞는 방향인지 알고 싶어서요."

[톤 규칙]
- 짧게 (3-4문장 이내)
- 상대방이 한 말을 구체적으로 언급 (개인화 필수)
- "리서치/실험" 프레이밍 (판매 프레이밍 금지)
- "No pitch" / "홍보 아니고" 명시
- 시간 약속 ("20분")으로 부담 최소화
- 거절해도 OK인 톤

[채널별 DM 주의사항]
- Discord: 같은 서버에서 최소 며칠 활동 후 DM
- Telegram: 같은 그룹 소속이면 바로 DM 가능
- Twitter: 먼저 상대방 트윗에 유의미한 답글 1-2회 → DM
- Reddit: Reddit DM은 응답률 낮음 → 글에 댓글로 먼저 접근
- GitHub: issue에 도움되는 댓글 먼저 → 프로필 이메일로 연락
```

### 5. 1:1 콜/세션 스크립트

```
DM 응답 후 실제 세션 진행 스크립트입니다.

[세션 구조 — 20분]

(0-3분) 인사 & 배경
- "감사합니다. 저도 {같은 문제}를 겪었고, {해결책}을 만들어보고 있어요."
- "오늘은 그냥 평소대로 작업하시면 되고, 제가 관찰하면서 질문 몇 개만 드릴게요."

(3-10분) 관찰 & 질문
- "평소 {작업}을 어떻게 하시나요?"
- "이 부분에서 불편한 점이 있나요?"
- "{문제 상황}이 발생하면 어떻게 대처하세요?"
- (메모: 그 사람의 실제 워크플로우, 언어, 불만 포인트)

(10-15분) 제품 시연 & 설치
- "제가 만들고 있는 게 딱 이 부분인데, 한번 보여드릴게요."
- 직접 설치해주기 (가능하면 화면 공유로)
- 그 사람의 워크플로우에 맞게 설정 조정
- "기본 설정은 이렇게 되어 있는데, {그 사람 패턴}에 맞게 이렇게 바꿔드릴게요."

(15-20분) 마무리 & 리텐션 설정
- "1주일만 이대로 써보시고 느낀 점 알려주시겠어요?"
- "불편한 점 있으면 바로 DM 주세요, 고칠게요."
- (핵심: 1주일 후 체크인 약속 잡기)

[세션 후 체크인]
- D+3: "어떠세요? 혹시 불편한 점 있었나요?"
- D+7: "1주일 써보셨는데 어떠셨어요? 계속 쓸 것 같으세요?"
- 피드백 → 즉시 반영 → "말씀하신 거 반영했어요" 메시지
```

### 6. 버려야 할 것 리스트

```
이 전략을 실행하면서 동시에 하면 안 되는 것:

[시간 낭비 — 초기에 버릴 것]
- 범용 서브레딧에 shotgun posting (vanity traffic만 발생)
- Product Hunt 런칭 (첫 10명 확보 전에는 시기상조)
- 콜드 이메일 대량 발송 (응답률 극히 낮음)
- 완벽한 랜딩 페이지 만들기 (10명한테는 DM이면 충분)
- 소셜 미디어 계정 꾸미기 (아직 팔로워 0명)

[나중에 해도 되는 것]
- 유료 광고 (PMF 확인 후)
- 인플루언서 아웃리치 (첫 10명의 피드백 반영 후)
- 프레스 릴리즈 (스토리가 만들어진 후)
- SEO (장기 전략, 병행은 OK)

> "첫 10명이 product-market fit의 증거다. 10명이 계속 쓰면 다음 단계로 간다.
> 10명도 유지 못하면 마케팅이 아니라 제품이 문제다."
```
