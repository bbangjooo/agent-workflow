# Analysis Framework

Step 3: 결과 분석 프레임워크

## 설명

실험 결과를 올바르게 해석하기 위한 분석 프레임워크를 사전에 설계하는 스킬입니다. 실험이 끝난 후가 아니라, 시작하기 전에 분석 방법을 확정하여 p-hacking을 방지합니다.

## 트리거

- Step 2 (experiment-plan) 완료 후 자동 진행

## 입력

- `hypothesis.md` — 가설 및 KPI
- `experiment-plan.md` — 실험 계획 (샘플 사이즈, 기간)

## 실행 내용

### 1. 통계 방법론 선택

```
빈도주의 (Frequentist) — 기본 권장:
- 장점: 해석이 명확, 업계 표준
- 단점: 중간 확인 제한, 고정 샘플 필요
- 적합: 트래픽 충분, 명확한 종료 시점

베이지안 (Bayesian) — 트래픽 부족 시 대안:
- 장점: 중간 확인 가능, "B가 A보다 나을 확률 X%"로 해석
- 단점: 사전 분포 설정 필요
- 적합: 트래픽 부족, 빠른 의사결정 필요

Sequential Testing — 조기 종료 필요 시:
- 장점: 미리 정한 시점에 중간 확인 가능
- 단점: 검정 횟수만큼 유의수준 보정 필요
- 적합: 리스크 높은 변경, 빠른 결론 필요
```

### 2. 검정 방법 설정

```
비율 비교 (전환율 등):
- Two-proportion z-test (큰 샘플)
- Fisher's exact test (작은 샘플)
- Chi-squared test (다변형)

연속 변수 (매출, 시간 등):
- Two-sample t-test (정규 분포)
- Mann-Whitney U test (비정규)
- Bootstrap confidence interval

검정 방향:
- 양측 검정 (기본): Treatment가 Control보다 좋거나 나쁠 수 있을 때
- 단측 검정: 한 방향만 관심 있을 때 (사전에 명시 필요)
```

### 3. 판단 기준 확정

```
통계적 유의성:
- p-value < {α} → 통계적으로 유의미한 차이 있음
- Confidence interval이 0을 포함하지 않음

실질적 유의성:
- 통계적으로 유의해도, 효과 크기가 의미 있는가?
- 최소 의미 있는 효과: {MDE}
- ROI 관점: 구현/유지 비용 대비 효과가 충분한가?

결과 판정 매트릭스:
┌──────────────┬──────────────────┬──────────────────┐
│              │ p < α            │ p ≥ α            │
├──────────────┼──────────────────┼──────────────────┤
│ 효과 ≥ MDE   │ ✅ 의미 있는 승리  │ ⏳ 샘플 부족 가능  │
│ 효과 < MDE   │ 📊 통계적만 유의  │ ➡️ 무승부         │
└──────────────┴──────────────────┴──────────────────┘
```

### 4. 시나리오별 대응 가이드

```
시나리오 A — Treatment 승리 (Primary KPI 개선, 유의):
  → 롤아웃 기준: 가드레일 지표도 악화 없음
  → 액션: Treatment를 100%로 롤아웃
  → 기록: 효과 크기, 학습점

시나리오 B — Control 승리 (Treatment가 오히려 나쁨):
  → 액션: Treatment 코드 제거, Control 유지
  → 기록: 왜 가설이 틀렸는지 분석, 학습점 정리

시나리오 C — 무승부 (Inconclusive):
  → 가능 원인: 샘플 부족, 실제로 차이 없음, MDE가 너무 작음
  → 액션 선택지:
    1. 실험 기간 연장 (샘플 부족이면)
    2. 더 큰 변경으로 새 실험 설계
    3. 현재 버전 유지하고 다른 영역 테스트

시나리오 D — 가드레일 위반:
  → 액션: 즉시 실험 중단, Control로 롤백
  → 기록: 어떤 가드레일이 왜 위반되었는지 분석
```

### 5. 분석 시 주의사항

```
Novelty Effect (새로움 효과):
- 변경이 "새로워서" 초기에 클릭이 많을 수 있음
- 대응: 실험 시작 후 첫 1-2일 데이터는 별도 분석

Primacy Effect (친숙함 효과):
- 기존 유저가 변경에 적응하느라 초기에 성과 저하 가능
- 대응: 신규 유저와 기존 유저 세그먼트 분리 분석

Simpson's Paradox:
- 전체 결과와 세그먼트별 결과가 반대일 수 있음
- 대응: 사전 정의한 세그먼트별 결과도 확인

Multiple Comparisons:
- Secondary KPI가 많으면 우연히 유의한 결과 나올 수 있음
- 대응: Primary KPI로만 의사결정, Secondary는 참고용
- 필요 시: Bonferroni correction 적용
```

### 6. 세그먼트별 분석 계획

```
사전 정의된 세그먼트만 분석 (experiment-plan.md 참조):
- {세그먼트 1}: {분석 목적}
- {세그먼트 2}: {분석 목적}

주의: 실험 후 결과를 보고 세그먼트를 "발굴"하는 것은 p-hacking
→ 의외의 세그먼트 차이 발견 시, 별도 실험으로 검증 필요
```
