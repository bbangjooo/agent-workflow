## 4.0 한 단락 요약 (TL;DR)

신·구 DB에 동시 쓰기 + 읽기는 구 DB 유지의 dual-write 모드를 7일 운용. 1.24M 신규 row 양쪽 0 mismatch (`shadow-diff` 자동 비교). M2 백필 단계 진입 자격 확보. 단, 정합성 검증이 *수동 1일 1회*라 회귀 감지 SLA 미흡.

## 4.1 왜 이 작업을 했나

- §북극성 "dual-write 정합성 100%" + "백필 진척도 진입 자격" 두 행을 trigger.
- 직전 phase에서 베이스라인 측정 완료 → §2 risk assessment에서 dual-write 진입 안전 조건 충족 (peak QPS 미커버지만 평일 평균 트래픽은 OK 판정).
- 대안: 직접 cutover(2.1 B안). 거부 사유 — 롤백 비용 ↑.

## 4.5 발견된 부수 이슈

- 정합성 검증 cron 미자동화 — 수동 1일 1회. 회귀 시 평균 12h 발견 지연.
- backfill rate-limit 정책 미정. DB load 영향 미측정.
- 정합성 스크립트가 실행 5분만 미가동 시 알람 없음 — 알람 추가 권장.

## 4.6 시스템 영향 분석 ★

- 이 phase 이전: 모든 쓰기 → 구 DB only. 신 DB는 schema만 존재.
- 이 phase 이후: 모든 쓰기 → 양쪽 DB. 신 DB가 *현재 시점부터의 진실*을 반영하기 시작.
- 실제로 가능해진 행동:
  - M2 백필을 *과거 데이터만* 대상으로 좁힐 수 있음 (현재 시점 이후는 dual-write가 자동 동기)
  - 신 DB로의 read swap을 단계적으로 시도할 수 있음 (canary 자격 확보)
- 실제로 불가능해진 행동: (없음 — 추가만 됨)
- downstream 영향:
  - 신 DB write QPS +200 (모니터링: `prometheus payments_v2_writes_total`)
  - 구 DB latency +3.2ms (가설 내, p95 800→803ms — *사용자 영향 측정 가능*)
  - downstream svc-orders는 무영향 (구 DB만 읽음)
- 사용자/외부 관찰자에게 보일까?: **아니오 (내부 인프라).** 단, latency +3.2ms는 *원칙적으로* 측정 가능. SLA 6% 내라 위배 없음 확인.

## 4.9 비관 재채점 — 이 phase 자체

> 적대적 리뷰어가 본다면 이 phase 산출물 중 무엇이 *과대평가*인가?
- "정합성 100%" — 1주는 짧음. 2-3주 또는 결제 마감일/월말 포함 검증이 더 robust. 1주 결과로 ✅ 마킹은 *낙관 채점*. 비관 채점 ○ 적정.
- "Latency +3.2ms 가설 내" — 가설 자체의 임계값(5%)이 임의. 사용자 SLA 기준이 아님 — 사용자 SLA 정의 자체가 부재.
- "Peak hour 1일치 OK" — 1일은 통계적 의미 없음. 월말/분기말 같은 *외란 이벤트* 일자 미커버.
