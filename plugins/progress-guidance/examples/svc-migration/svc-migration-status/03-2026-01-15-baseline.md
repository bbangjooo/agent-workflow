## 3.0 한 단락 요약 (TL;DR)

미그레이션 시작 전 *현재 시스템의 정량 상태*를 측정. p95 latency, monthly downtime, DB row count, lock 빈도 4개 메트릭 베이스라인 확정. **infrastructure-only phase** — §북극성 row 자체는 안 움직이지만, *측정 가능* 상태로 진입한 것이 의미.

## 3.1 왜 이 작업을 했나

- §북극성 표를 *측정값*으로 채우려면 현재값이 필요.
- 사용자 목표 "p95 50% 감소"는 800ms 기준이라는 합의가 *측정 없이는 임의*.
- 직전 cycle: 사용자가 미그레이션 의사결정 (§2.1 dual-write A안) 합의. 다음 단계는 측정 가능 상태 확보.

## 3.2 무엇을 만들었나

| 작업 | 위치 | LOC | 검증 산출물 |
|---|---|---|---|
| metrics 수집 스크립트 | `ops/baseline/measure.sh` | 80 | `metrics-baseline.json` (1주치 raw) |

## 3.3 검증 (근거)

- 단위 테스트: 없음 (shell script).
- 실 데이터 검증: `bash ops/baseline/measure.sh > metrics-baseline.json` → JSON에 4 메트릭 평균/95퍼센타일 포함.
  - p95 latency: 803ms (1주 평균)
  - monthly downtime: 30min (incident log 5건 평균)
  - DB row count: 1,231M orders + 412M payments_legacy
  - lock 빈도: 평균 13건/일
- 외부 검증: incident log 5건 수동 spot-check, downtime 합계 일치.

## 3.4 결과 vs 가설

| 가설 | 실제 측정값 | 차이 사유 |
|---|---|---|
| p95 ~700ms | 803ms | 가설보다 12% 높음 — peak hour 미고려했었음 |
| downtime ~20min/월 | 30min | DB lock 사건이 가설보다 빈번 |
| row count ~1B | 1.23B (orders only) | 추정 누락 |

## 3.5 발견된 부수 이슈

- prometheus retention이 7일밖에 안 되어 1년 추세 분석 불가. 별도 작업 필요 (별 trigger 없으면 미루기).
- incident log 형식이 표준화 안 되어 root-cause classification 수동.

## 3.6 시스템 영향 분석 ★

- 이 phase 이전: 시스템 변화 없음 (측정 도구만 추가).
- 이 phase 이후: 시스템 변화 없음 (측정 결과는 doc에만).
- 실제로 가능해진 행동:
  - §북극성 표의 4개 row 중 3개에 `현재` 값 채워짐 → 갭 절대치 처음으로 *수치로* 표현 가능
  - 향후 phase의 결과를 "↓ 12%" 처럼 *상대* 비교 가능
- 실제로 불가능해진 행동: (없음)
- downstream 영향:
  - prod 시스템 직접 영향 0 (read-only 측정)
  - 매주 1회 measure.sh 실행 → DB read load +0.5% (모니터링: `prometheus measure_load`)
- 사용자/외부 관찰자에게 보일까?: **아니오**.

> **이 phase는 infrastructure-only.** §북극성 row의 `현재` 값을 *측정 가능*하게 한 것이 의미. 이게 없으면 이후 phase의 모든 progress 주장은 가설 vs 가설 비교가 됨. 이 phase가 enables: §4 (dual-write의 latency overhead 측정), §5 (백필 진척도 추적), §6 (read swap 후 p95 비교).

## 3.7 §북극성 갱신 (이 cycle 이후)

- 모든 row의 `현재` 값이 비어 있었음 → 측정값 채워짐.
- 갭 절대치 처음으로 수치 표현 가능: p95 -50%, downtime -30min, 백필 -100%.

## 3.8 §pipeline 매핑 영향

- §1 Inventory: ✗ → ◎. 근거: row count + schema 목록 export 완료 (`metrics-baseline.json`).
- §2 Risk assessment: ✗ → ○. 근거: incident log 분석 (5건). 단, peak QPS 시 risk는 미측정 → 비관 채점 ○.

## 3.9 비관 재채점 — 이 phase 자체

> 적대적 리뷰어가 본다면 이 phase의 *과대평가* 포인트는?
- 1주 측정 → 월말/분기말 등 *외란 이벤트* 미커버. peak QPS 일자가 측정 기간에 포함됐는지 확실치 않음.
- prometheus retention 7일이라 *통계적 안정성* 측정 부족 — 1주 평균이 진짜 평균인지 1주 fluke인지 모름.
